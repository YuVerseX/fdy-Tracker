# 后台任务软取消设计方案

## 背景

当前后台长任务已经具备：

- 统一的任务记录落点：`admin_task_runs.json`
- 统一的任务摘要与任务中心展示
- 基于 `progress_callback` 的运行中指标回传
- 任务互斥控制，避免多类内容变更任务并发写入

但系统仍缺少一个明确的“提前终止”能力。用户一旦误点大批量任务、发现参数设置不合适，或者临时需要切换到别的处理任务，只能等待当前任务自己跑完。

这不是单纯的前端交互缺口，而是一个跨层能力缺口：

- 前端没有“终止任务”按钮和状态表达
- 后端没有取消接口
- 任务记录没有“已请求终止 / 已终止”语义
- 业务服务没有协作式中断检查点

## 核心判断

本期应实现 **软取消（cooperative cancellation）**，不做强杀和回滚。

也就是说：

- 已处理完成并已落库的结果保留
- 当前处理单元允许自然收尾
- 停止后续未开始的处理单元
- 任务最终状态记为 `cancelled`

这是与当前架构最匹配、风险最低的取消模型。

## 问题定义

当前长任务具有以下共性：

### 1. 任务是后台异步执行，不是可直接 kill 的独立 worker

管理台任务由 `src/api/admin.py` 通过 `BackgroundTasks` 启动，再调用各自服务函数执行。当前没有统一 job runner、队列系统或可中断 executor。

### 2. 多个任务采用“逐条处理 + 分批提交”模式

例如：

- `run_ai_analysis()` 按帖子逐条分析
- `backfill_post_jobs()` 按帖子逐条抽岗位
- `backfill_existing_attachments()` 按帖子逐条补附件与字段
- `run_duplicate_backfill()` 按候选集和重复组逐步推进

这些任务通常在循环中使用 `db.begin_nested()`，最后统一 `commit()`。这意味着强行中断线程并不安全。

### 3. 部分任务包含较慢的外部调用

例如抓取详情页、下载附件、调用 OpenAI。这些操作可能在单个处理单元内部耗时较长，因此取消不可能做到“点击即停”。

## 目标

- 为后台任务中心增加真实可用的“提前终止”能力
- 保证取消后的任务状态、文案、后续可执行操作都清晰可理解
- 保持已处理结果可复用，不因取消而破坏现有数据
- 尽量复用现有任务记录、进度回调、任务卡片体系

## 实现状态（2026-04-01）

Phase 1 已落地并完成验证，当前实际覆盖：

- `attachment_backfill`
- `base_analysis_backfill`
- `ai_analysis`
- `job_extraction`
- `duplicate_backfill`

本轮仍未纳入：

- `manual_scrape`
- `scheduled_scrape`

也就是说，任务中心现在只会为上述内容处理类任务暴露“提前终止”，不会为实时抓取任务暴露取消按钮。

## 非目标

- 本期不做强制线程终止
- 本期不做事务级全量回滚
- 本期不引入外部队列系统或 worker 框架
- 本期不改变现有任务互斥矩阵
- 本期不把抓取类任务与内容处理类任务一起做完

## 设计原则

### 1. 先保证数据安全，再追求“停止得更快”

取消是为了避免继续扩大处理范围，不是为了抹掉已经产生的结果。

### 2. 取消必须是用户可理解的状态，而不是隐藏内部标记

用户需要区分：

- 任务仍在运行
- 任务已经收到终止请求，正在收尾
- 任务已终止

### 3. 只在安全边界检查取消

取消检查点只放在：

- 进入下一条帖子前
- 进入下一组重复候选前
- 进入下一页抓取前

不在单条帖子或单次外部调用的中间硬切。

### 4. 已终止不等于失败

`cancelled` 表示“用户主动停止，系统正常收尾”，不应使用失败态文案误导用户。

## 范围分期

### Phase 1：先支持内容处理类任务

本期纳入：

- `attachment_backfill`
- `base_analysis_backfill`
- `ai_analysis`
- `job_extraction`
- `duplicate_backfill`

### Phase 1.5：抓取类任务单独评估

暂不纳入：

- `manual_scrape`
- `scheduled_scrape`

原因：

- 抓取流程与外部网站请求绑定更深
- 单个页面抓取、详情抓取和附件发现的取消粒度与其他任务不同
- 需要单独评估“停止在半页 / 半条抓取过程”时的体验与一致性

## 状态模型

### 一、任务状态

新增状态：

- `cancelled`

状态标签建议为：

- `running` -> `执行中`
- `failed` -> `失败`
- `success` -> `完成`
- `cancelled` -> `已终止`

### 二、取消请求中间态

不新增独立顶层状态 `cancelling`，而是保持：

- 顶层 `status = running`
- 详情字段里增加 `cancel_requested_at`
- 前端据此显示“正在终止”

这样可以避免把运行中任务过早从互斥判定里释放出去。

### 三、任务详情字段

任务记录新增可选字段：

- `cancel_requested_at`
- `cancel_reason`
- `cancel_requested_by`

展示层兼容字段：

- `cancel_requested`：前端可由 `Boolean(cancel_requested_at)` 推导

### 四、最终取消结果

当任务在检查点检测到取消请求后，最终记录为：

- `status = cancelled`
- `phase = 已终止`
- `summary = 用户已提前终止，本次已保留已处理结果`
- `finished_at` 写入实际终止时间

若任务已累计真实 metrics，summary 追加结果摘要，例如：

- `用户已提前终止，已处理 18 条，已写入 42 条岗位`

## API 设计

新增接口：

- `POST /api/admin/task-runs/{task_id}/cancel`

行为约束：

### 1. 仅运行中任务可提交取消

若任务不是运行中：

- 返回 `409`
- 错误文案明确说明“当前任务已经结束，不能再终止”

### 2. 已提交过取消请求时允许幂等返回

若同一任务已经有 `cancel_requested_at`：

- 返回 `202`
- message 为“终止请求已提交，正在等待当前处理单元结束”

### 3. 成功提交取消请求

返回：

- `message`
- 最新序列化后的 `task_run`

## 后端执行模型

### 一、任务记录服务

在 `src/services/admin_task_service.py` 增加：

- `request_task_run_cancel(task_id, cancel_reason=None, cancel_requested_by=None)`
- `is_task_run_cancel_requested(task_id)`

其中：

- `request_task_run_cancel()` 负责把取消请求写入任务记录
- `is_task_run_cancel_requested()` 供长任务运行时轮询

### 二、取消异常

在 `src/services/task_progress.py` 增加：

- `TaskCancellationRequested`
- `CancelCheck`
- `raise_if_cancel_requested(cancel_check)`

目的：

- 让业务服务不直接依赖任务记录文件
- 只依赖一个轻量的可调用检查器

### 三、后台包装层

在 `src/api/admin.py` 中：

- 为每个长任务构造 `cancel_check`
- 把 `cancel_check` 传给服务函数
- 捕获 `TaskCancellationRequested`
- 统一写 `record_task_run(... status="cancelled" ...)`

## 业务服务取消检查点

### 一、AI 分析

在 `run_ai_analysis()` 中：

- 每次准备处理下一条帖子前先检查取消
- 当前帖子分析完成后再进入下一条

### 二、岗位抽取

在 `backfill_post_jobs()` 中：

- 每次准备处理下一条帖子前先检查取消

### 三、附件补处理

在 `backfill_existing_attachments()` 中：

- 每次准备处理下一条帖子前先检查取消

### 四、基础分析补齐

在 `backfill_base_analysis()` 中：

- 每次准备处理下一条帖子前先检查取消

### 五、重复治理

在 `run_duplicate_backfill()` 及其核心循环中：

- 每次进入下一组候选前检查取消
- 每次进入下一组重复组写入前检查取消

## 前端交互设计

### 一、任务卡按钮

对运行中任务新增操作：

- `提前终止`

按钮行为：

- 点击后进入 `提交中...`
- 请求成功后刷新任务中心和任务摘要

### 二、运行中但已提交终止请求

界面表达：

- 状态仍视为运行中
- 额外 notice：`终止请求已提交，当前处理单元结束后会停止`
- “提前终止”按钮禁用

### 三、已终止任务

界面表达：

- Badge：`已终止`
- 提示文案：`这次处理由你手动终止，已完成部分结果已保留`
- 结果区继续显示已处理计数

### 四、后续操作

本期不新增新的动作 key。

已终止任务先沿用现有语义：

- 默认保留 `按原条件重试`
- 对支持增量补齐的任务，额外保留 `只补剩余`
- 不默认展示 `再次运行`

这样可以让用户在终止后继续补缺，而不用再新增“继续处理”专属动作。

## 风险与权衡

### 1. 取消不是即时生效

如果当前任务正卡在：

- 单次 OpenAI 调用
- 单个详情页抓取
- 单个附件下载

取消会在该单元完成后才生效。这是软取消的正常边界。

### 2. 已终止任务的 summary 需要避免和失败混淆

如果文案写得像失败，会让用户误以为系统异常。必须强调“主动终止”和“已保留结果”。

### 3. 互斥释放时机不能过早

只要任务仍是 `running`，即使已请求终止，也不能让其他互斥任务启动。必须等最终落成 `cancelled`。

## 验证标准

当本期功能完成后，应满足：

### 后端

- 运行中任务可以提交取消请求
- 已结束任务不能被取消
- 取消请求会写入任务记录
- 长任务能在安全检查点停止
- 终止后任务最终状态为 `cancelled`

### 前端

- 运行中任务卡显示“提前终止”按钮
- 已请求终止时任务卡显示“正在终止”提示
- 已终止任务显示“已终止”状态，不误报为失败
- 已终止任务仍能看到已处理结果

## 结论

本期应实现的是：

`以任务记录为中心的协作式软取消，而不是线程级强杀。`

它的核心收益不是“瞬间停下”，而是：

- 让用户可以及时止损
- 保住已经处理出的结果
- 让后台状态继续真实、可解释、可追溯
