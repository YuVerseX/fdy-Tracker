# 招聘详情与管理台重设计方案

## 背景

当前项目已经具备可演示、可本地使用的主链路，但前台详情页和后台管理台仍明显停留在“功能拼装完成”的阶段，没有形成稳定的产品表达。

当前问题不是单纯的“界面不够好看”，而是三个层面同时失稳：

- 页面文案混入了设计说明、实现约束和内部术语
- 任务进度、计数和重试语义不真实，导致后台状态不可预测
- 缺少共享组件层和统一设计 token，页面只能靠局部 Tailwind 拼接

这次重设计不以新增功能为目标，而是以“让已有能力更清楚、更可信、更一致”为目标，先把产品骨架立稳。

本 spec 用于覆盖和替代此前偏“数据治理拆页”的局部设计方向；如与旧文档冲突，以本 spec 为准。

## 核心问题

### 1. 最终用户页面暴露了内部视角

招聘详情页和管理台里出现了大量面向设计者或开发者的表达，例如：

- “按任务目标拆分入口，避免把基础处理和 AI 增强混在一起”
- “默认先看当前异常和最近结果，历史明细收进下方折叠区”
- “岗位级结果 / 原始字段”
- “规则分析 / OpenAI 分析”
- “当前还没开放岗位摘要接口，先显示占位值”

这些文案在设计说明、测试断言和评审文档里是合理的，但不应直接出现在最终界面中。

### 2. 后台任务状态看起来像实时，实际上只是阶段占位

当前进度条使用固定阶段值表达任务过程：

- 手动抓取：`10 -> 55 -> 90 -> 100`
- 附件补处理：`10 -> 60 -> 90 -> 100`
- 基础分析 / AI 分析 / 岗位抽取：`10 -> 65 -> 90 -> 100`

这意味着页面显示的百分比并不代表真实完成比例，而只是“当前大致在哪一阶段”。这会直接损害后台可信度。

### 3. 任务计数模型和展示模型没有对齐

运行中的任务大多只更新：

- `phase`
- `progress`
- `heartbeat`

最终计数通常在任务结束后一次性写入。因此用户在运行中看不到“已处理多少条 / 新增多少条 / 下载多少附件”的真实变化。

更糟的是，前端把某些不存在的指标直接渲染为 `0`，会让用户误判为“确实处理了 0 条”。

### 4. 重试语义没有产品定义

当前“重试任务”本质上只是拿上一次参数重新发起同类任务，而不是有明确定义的“失败重试 / 再次执行 / 补跑剩余”。这会让已完成任务、失败任务和运行中任务在按钮语义上混在一起。

### 5. 详情页信息结构过重

招聘详情页当前把：

- 关键信息
- 分析结果
- 岗位级结果
- 附件
- 正文

都做成较重的大块卡片，导致页面更像内部分析面板，而不像面向最终用户的公告阅读页。

### 6. 缺少共享组件与统一设计系统

当前项目几乎没有公共 UI primitives，导致：

- 间距节奏在各页漂移
- 状态块和提示块样式不统一
- 色彩语义混杂
- 大文件承载过多业务判断和样式决策

## 目标

- 所有最终用户可见文案统一改为用户视角表达
- 后台任务状态表达必须“语义真实”，不能再把阶段占位伪装成真实百分比
- 详情页改造成“阅读与决策页”，而不是“系统分析页”
- 后台改造成“任务导向后台”，而不是“实现流程说明页”
- 建立最小但稳定的组件层和设计 token
- 为后续实现 plan 提供稳定边界，避免边改边重新定义产品

## 非目标

- 本期不新增新的抓取源
- 本期不重做整套权限系统
- 本期不引入复杂监控平台
- 本期不重做数据库 schema
- 本期不追求像企业级 BI 一样的高级图表能力
- 本期不把后台改造成多角色运营系统

## 设计原则

### 1. 只表达用户需要知道的事

页面只回答：

- 现在是什么状态
- 我能做什么
- 下一步建议做什么

不直接暴露：

- 开发说明
- 接口占位
- 环境变量名
- SDK 是否安装
- provider / base_url / 内部实现标签
- “岗位级结果优先回退原始字段”这类实现策略说明

### 2. 先保证真实性，再谈精细感

如果当前只能知道“阶段”，就明确显示阶段；
如果当前只能知道“心跳还活着”，就明确显示任务仍在运行；
如果数量还未统计完成，就显示“统计中”而不是先填 `0`。

### 3. 公共页强调阅读效率，后台页强调操作可信度

- 前台详情页优先做“快读”和“判断是否值得关注”
- 后台优先做“当前状态可理解、操作结果可预测”

### 4. AI 是增强层，不是页面语义中心

AI 在系统里是增强能力，不应成为后台所有操作的中心叙事。页面应先成立“基础处理可单独工作”的心智模型，再追加 AI 能力。

### 5. 用结构减少无效解释

优先通过：

- 分组
- 标签
- 状态色
- 紧凑字段布局
- SVG 图标

来降低说明性文案，而不是反复用整句文字解释页面结构。

## 信息架构

### 一、后台一级结构

推荐把后台一级导航收敛成 4 个入口：

- `总览`
- `处理任务`
- `任务中心`
- `系统设置`

不再把 `AI 增强` 单独做成一级入口。

原因：

- 当前一级导航已经承载了过多中间概念
- `AI 增强` 本质属于“处理任务”的一个子模式
- 用户先找“我要做什么”，再找“这一步是否使用 AI”，比先分到独立 AI 页更自然

### 1. 总览

只回答两个问题：

1. 当前系统整体是否健康
2. 接下来最值得处理的积压是什么

只保留：

- 系统状态摘要
- 最近成功 / 最近失败
- 当前正在运行的任务
- 待处理积压摘要

不放复杂表单和长说明。

### 2. 处理任务

作为主操作页，但内部只分为两个二级段落：

- `基础处理`
- `AI 增强`

#### 基础处理

包含 4 个任务卡组：

- 抓取与附件
- 重复治理
- 内容分析
- 岗位索引

#### AI 增强

包含 2 个任务卡组：

- 内容增强
- 难岗位补抽

这样既保留“基础与 AI 分层”，又避免再新增一个一级导航。

### 3. 任务中心

专门负责“任务事实”，不承担业务解释。

内部结构固定为：

- 当前任务
- 最近结果
- 历史记录

每条任务卡片只表达：

- 做什么
- 进行到哪一步
- 当前处理量
- 可执行操作

### 4. 系统设置

承载：

- 定时抓取配置
- 默认抓取源
- 运行环境状态摘要

不混入业务操作入口。

### 二、前台详情页结构

详情页从“分析卡片堆叠”改为“阅读页”。

推荐结构：

1. 标题区
2. 公告元信息
3. 关键信息摘要
4. 岗位明细
5. 附件
6. 正文
7. 信息说明（折叠）

### 1. 标题区

包含：

- 标题
- 发布日期
- 来源
- 公告类型或辅导员相关标签

移除：

- 面向后台的快捷入口
- “匹配度”这类内部评分

### 2. 关键信息摘要

不再使用当前的大块卡片阵列，而改成紧凑 2 列事实区。

字段优先级：

- 招聘岗位
- 招聘人数
- 学历要求
- 专业要求
- 工作地点
- 政治面貌
- 年龄要求
- 报名时间

设计要求：

- 单项高度统一
- 更接近 definition list，而不是 KPI 卡片
- 多数字段不需要强调色背景
- 仅对缺失、待确认、重要状态做轻量标签

### 3. 岗位明细

如果只有 1 个岗位：

- 用紧凑字段块展示

如果有多个岗位：

- 直接使用表格

表格列建议：

- 岗位名称
- 人数
- 学历
- 专业
- 地点
- 备注

不再默认显示“来源: 附件表格 / 正文字段 / AI 抽取”这类内部来源标签。来源信息如果需要保留，应折叠进“信息说明”。

### 4. 信息说明

折叠区承载：

- 数据来自正文 / 附件 / 系统整理
- 最近抓取时间说明
- 结构化结果可能与原文存在轻微归纳差异

这部分是说明区，不应占据首屏。

## 状态模型设计

### 一、统一任务展示模型

后台任务接口应统一输出以下展示模型：

- `id`
- `task_type`
- `display_name`
- `status`
- `status_label`
- `progress_mode`
- `stage_key`
- `stage_label`
- `started_at`
- `heartbeat_at`
- `finished_at`
- `duration_ms`
- `scope_summary`
- `rerun_of_task_id`
- `metrics`
- `failure_reason`
- `actions`

其中：

- `display_name` 面向用户
- `stage_label` 面向用户
- `metrics` 用于任务运行中实时计数
- `actions` 明确前端当前允许显示哪些操作

### 二、进度模式

只允许两种模式：

### 1. `stage_only`

适用于抓取总量未知或动态变化的任务。

页面表现：

- 不显示确定百分比
- 显示当前阶段
- 显示实时计数

例如：

- 已扫描 1 页
- 已发现 24 条公告
- 新增 8 条
- 更新 5 条

### 2. `determinate`

适用于总量已知的任务。

页面表现：

- 显示 `completed / total`
- 百分比由 `completed / total` 计算

例如：

- 已处理 46 / 100 条

### 三、运行中实时计数

以下任务需要支持运行中 metrics：

### 手动抓取 / 定时抓取

- `pages_scanned`
- `posts_seen`
- `posts_created`
- `posts_updated`
- `attachments_discovered`

### 附件补处理

- `posts_scanned`
- `posts_updated`
- `attachments_downloaded`
- `attachments_parsed`
- `fields_added`

### 基础分析 / AI 增强

- `posts_scanned`
- `posts_updated`
- `analysis_created`
- `insight_created`
- `fallback_count`

### 岗位索引 / AI 岗位补抽

- `posts_scanned`
- `posts_updated`
- `jobs_saved`
- `ai_posts`

如果某项指标当前任务并不产生，就不展示；不要用 `0` 占位冒充真实结果。

### 四、重试与再运行语义

任务操作应明确分成三类：

- `按原条件重试`
- `再次运行`
- `只补剩余`

规则：

- 运行中任务：不显示重试
- 失败任务：显示 `按原条件重试`
- 成功任务：显示 `再次运行`
- 支持补缺口的任务：额外显示 `只补剩余`

所有重试都会生成新任务，不覆盖旧任务。

新增字段：

- `rerun_of_task_id`

用于让历史记录可追溯。

### 五、任务冲突规则

当前所有内容变更任务几乎全互斥，虽然安全，但会让后台显得“什么都不能同时做”。

本期先不改变后端互斥策略，但页面需要如实表达：

- 当前有互斥任务运行中
- 某些动作暂时不可执行
- 原因是“为了避免重复写入和状态冲突”

这部分说明应在按钮禁用态上表达，不要放成大段顶部提示。

## 组件与设计系统

### 一、基础设计 token

建立一套最小 token：

- 颜色语义
  - 信息：蓝
  - 成功：绿
  - 警示：橙
  - 错误：红
  - 中性：灰
- 间距
  - 8 / 12 / 16 / 24 / 32
- 圆角
  - 8 / 12 / 16
- 字号层级
  - 页面标题
  - 区块标题
  - 字段标签
  - 正文
  - 辅助说明

不要再让业务页面自行决定同类块的 spacing 和 tone。

### 二、共享组件层

需要抽出的共享组件至少包括：

- `PageHeader`
- `SectionHeader`
- `StatusBadge`
- `NoticeBanner`
- `FactList`
- `MetricStrip`
- `TaskRunCard`
- `EmptyState`
- `InfoDisclosure`
- `ActionButtonGroup`

这些组件应先统一交互和视觉，再承载业务内容。

### 三、组件库策略

推荐引入 `Reka UI` 作为无样式 primitives 层，不直接引入重样式组件库。

使用边界：

- 适合：`Tabs`、`Accordion`、`Dialog`、`Select`、`Collapsible`、`Tooltip`、`Progress`、`Toast`
- 不适合：直接拿来决定业务卡片和页面布局

原因：

- 当前项目最缺的是可访问、可组合的基础交互层
- 业务布局和内容密度仍需要自己掌控
- `Reka UI` 更适合当前 Vue + Tailwind 结构

参考：

- https://reka-ui.com/docs/overview/introduction
- https://reka-ui.com/docs/guides/migration

## 文案规范

所有最终界面遵循以下规则：

- 只写用户可理解的结果和动作
- 不暴露实现策略
- 不暴露接口状态和开发说明
- 不出现 TODO、FIXME、mock、stub、debug、待接接口、开发者备注
- 尽量避免“基础链路 / provider / source_id / sdk / runtime”这类工程术语直接出现在主界面

推荐文案方向：

- “当前正在抓取最新公告”
- “已处理 18 条，仍在继续”
- “最近一次成功更新时间”
- “按原条件重试”
- “再次运行”
- “仅补未完成内容”

不推荐文案方向：

- “默认先看当前异常和最近结果”
- “避免把基础处理和 AI 增强混在一起”
- “后端还没开放接口”
- “AI 增强开关还没有开启，基础模式仍可继续补齐”

## 风险与权衡

### 1. 需要先改任务接口，再改 UI

如果只改前端页面而不补状态字段，后台仍会继续出现“看起来很完整、实际信息很空”的问题。

### 2. 会推翻一部分已有页面组织

当前管理台已经按旧设计文档拆成多个页面区块。本次设计会对一级导航和二级结构做重新收束，需要接受一定的返工。

### 3. 需要补共享组件层

如果继续直接改页面文件，会再次形成大文件和重复样式。本次重构必须接受“先抽 UI primitives，再改页面”的成本。

## 验证标准

当本次设计落地后，应满足以下验收标准：

### 前台详情页

- 首屏不再出现内部来源说明和实现说明
- 关键信息区能在短时间内读完
- 多岗位公告不再以大卡片堆叠展示
- 信息说明被降级到折叠区

### 后台任务中心

- 运行中任务不再显示伪百分比
- 运行中能看到真实阶段和实时计数
- 失败、成功、运行中三类任务的按钮语义不同
- 已完成任务再次运行时，用户知道会发生什么

### 设计系统

- 同类按钮、提示块、状态标签和区块标题风格一致
- 页面不再出现设计说明型文案
- 不再用 `0` 去占位不存在的运行指标

## 分阶段实施建议

推荐分 4 个实现阶段：

### 阶段 1：状态模型收口

- 改任务接口与任务记录展示模型
- 加入实时 metrics
- 定义 retry / rerun / incremental run 语义

### 阶段 2：共享组件层

- 建立 UI primitives
- 引入 `Reka UI`
- 收敛 token 与状态语义

### 阶段 3：后台重做

- 重做总览
- 重做处理任务页
- 重做任务中心

### 阶段 4：前台详情页重做

- 调整信息层级
- 改关键信息区
- 重做多岗位展示
- 降级系统说明

## 结论

本次不应继续对现有页面做零散修补，而应执行一次“状态模型 + 信息架构 + 组件层”的联合收口。

这次重构的核心判断只有一句话：

`先让系统说真话，再让页面变好看。`
