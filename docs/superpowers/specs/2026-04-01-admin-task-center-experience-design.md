# 后台任务中心体验重构设计

## 背景

当前项目已经具备：

- 后台任务异步提交与任务记录
- `phase / progress / heartbeat_at` 基础状态能力
- 后台任务中心页面
- 公开前台 freshness 成功快照

但从真实体验看，这一套能力还没有形成稳定的产品表达。

当前问题不在于“有没有任务中心”，而在于：

- 后台页面对“实时感”的承诺强于实际可观测进度
- 抓取任务前半段缺少可见进展，容易让人误判为卡住
- 公开前台和后台管理页没有形成统一心智
- 少数加载态、超时态和兼容态文案会制造“伪实时”或“伪失败”感

本设计不追求把系统做成复杂的监控平台，而是把现有任务能力收口成一套：

- 后台可信
- 前台克制
- 状态真实
- 文案短而准

## 核心判断

### 1. 后台和前台不应该追求同一种“实时”

- 后台任务中心负责解释过程
- 公开前台负责展示最近一次成功快照

后台回答：

- 现在在做什么
- 做到哪一步
- 为什么还没看到结果数
- 现在该等、终止还是重试

前台回答：

- 现在看到的是不是最近一次成功更新后的结果
- 最近一次成功完成于什么时候

### 2. 本期不为公开前台增加运行态

对当前项目来说，公开前台是招聘公告阅读页，不是任务监控页。

用户的主要目标是：

- 找公告
- 看当前可用结果
- 判断是否值得继续查看

因此本期坚持：

- 前台继续只展示成功快照
- 不展示运行中任务
- 不轮询后台运行态
- 不展示半成品结果

### 3. 后台不应继续做“看起来像实时”的界面

后台需要的不是更花哨，而是更诚实：

- 当前阶段是什么
- 当前阶段能看到哪些指标
- 不能看到哪些指标
- 页面多久会自动同步一次

如果当前只有阶段，就明确表达阶段；
如果当前阶段没有结果计数，就明确表达“结果将在后续阶段出现”；
不再用强承诺文案暗示“数字马上会变”。

## 审计结论

## 当前真实行为

### 1. 后台任务中心是轮询，不是推送

- 管理页在存在运行中任务时按固定间隔轮询状态
- 长任务通过 heartbeat 维持“仍在运行”的信号
- 页面不是秒级实时，而是准实时

### 2. 抓取任务前半段天然缺少结果类指标

当前抓取链路先采集原始结果，再进入逐条保存。

这意味着：

- 前半段可以知道“正在采集”
- 但未必能立刻知道“新增多少条 / 更新多少条”

因此“阶段在动但结果数不动”在现实现状下是正常现象，不应再被页面包装成异常。

### 3. 公开前台当前是成功快照模型

公开 freshness 只关注最近一次抓取成功时间。

这个方向是对的，应继续保留。

## 发现的相邻问题

### 1. 前台存在“伪运行中”文案

前台列表和详情页在 freshness 请求加载时会显示“正在更新最近抓取记录”，
但这只是本地在读取 freshness，并不等于后台真的有抓取任务正在运行。

这会制造不必要的实时错觉。

### 2. 后台提交超时语义不够准确

请求超时时，当前前端反馈更接近 error，但真实语义通常是：

- 后台可能已接单
- 前端没等到响应
- 应刷新任务中心确认状态

这不应继续用“错误反馈”表达。

### 3. 兼容层过宽，掩盖契约漂移

前端对 freshness 和任务状态兼容了多种历史字段名。

这有利于过渡，但也会让：

- 接口真实契约不清晰
- 旧语义残留不容易暴露

本期应收口主契约，并把兼容层视为临时迁移措施。

## 目标

- 明确后台和前台的不同职责
- 让后台任务中心形成可信的过程体验
- 让公开前台继续保持成功快照语义
- 让任务状态、阶段、结果、异常反馈使用统一规则
- 用更少但更准的文案替代当前的强承诺表达

## 非目标

- 本期不为公开前台增加运行态
- 本期不引入 WebSocket / SSE 作为第一步
- 本期不做多数据源调度体系重构
- 本期不把后台做成通用作业平台或 BI 监控页
- 本期不新增复杂图表和监控告警系统

## 设计原则

### 1. 后台解释过程，前台展示结果

后台讲过程事实；
前台讲成功快照。

### 2. 默认少字，只在必要时解释

正常状态靠结构理解：

- badge
- 阶段轨迹
- 时间
- 结果数字
- 主按钮

只有异常态或当前阶段天然没有结果数字时，才补一行短说明。

### 3. 未知值不能伪装成确定值

- 不存在的指标不是 `0`
- 无法确认的进度不是“假百分比”
- 只是加载 freshness，不等于“后台正在更新”

### 4. 先把状态契约做真，再优化传输方式

本期优先统一状态语义与页面表达，
推送机制留到后续增强阶段。

### 5. 页面要自解释，不要写成说明书

复杂度应该藏在：

- 状态机
- 契约
- 信息层级

而不是堆到用户要阅读的大段说明文字里。

## 一、前台策略

### 1. 继续保留成功快照模型

公开前台只展示：

- 最近一次抓取成功时间
- 最近一次抓取成功标签
- freshness 不可用时的兜底文案

### 2. 本期不做这些能力

- 不展示运行中任务
- 不轮询后台运行态
- 不显示抓取阶段
- 不显示运行中计数
- 不在抓取半途中提前切换列表结果

### 3. 文案收口规则

前台 freshness 只允许使用以下语义：

- 正在读取最近抓取记录
- 最近一次成功完成于……
- 最近抓取记录暂时不可用
- 还没有可展示的抓取成功任务记录

禁止继续使用：

- 正在更新最近抓取记录
- 最近内容已更新

除非页面真的观测到了后台运行中的抓取任务。

## 二、后台任务状态契约

### 1. 统一任务状态与阶段

任务生命周期拆成两层：

- `status`：表达任务现在属于哪一类生命周期状态
- `stage`：只表达运行中的处理阶段

`status` 统一收口为：

- `queued`
- `running`
- `cancel_requested`
- `success`
- `failed`
- `cancelled`

说明：

- `queued`：任务已提交，等待后台真正开始
- `running`：任务正在执行
- `cancel_requested`：已收到终止请求，等待当前处理单元安全收尾
- `success`：任务正常完成
- `failed`：任务未完成且需要人工关注
- `cancelled`：任务已按终止请求结束

`stage` 只在 `queued / running / cancel_requested` 时使用，统一收口为：

- `submitted`
- `collecting`
- `persisting`
- `finalizing`

说明：

- `submitted`：任务已提交，正在等待后台开始或刚进入执行
- `collecting`：正在采集原始数据或扫描输入范围
- `persisting`：正在写入数据库并生成可见结果
- `finalizing`：正在做收尾整理

`success / failed / cancelled` 是终态，不再作为 `stage` 使用。

### 2. 运行态与完成态分离

运行态只表达当前已知事实：

- `status`
- `stage`
- `started_at`
- `stage_started_at`
- `heartbeat_at`
- `live_metrics`

完成态才表达最终结果：

- `finished_at`
- `duration_ms`
- `final_summary`
- `final_metrics`
- `failure_reason`

`stuck` 不作为后端持久化状态，而是前端基于 `heartbeat_at` 超过阈值后计算出的展示诊断态。

### 3. live_metrics 必须按阶段约束

不同阶段允许的指标不同。

例如抓取任务：

- `collecting`
  - `pages_fetched`
  - `detail_pages_fetched`
  - `raw_items_collected`
- `persisting`
  - `posts_seen`
  - `posts_created`
  - `posts_updated`
  - `failures`
- `finalizing`
  - 与收尾相关的补充指标

### 4. 缺失值语义

- 字段不存在：当前阶段尚不可得
- 字段存在且为 `0`：已经统计过，结果确实为 `0`

### 5. duration 规则

- `duration_ms` 只在终态写入
- 运行中耗时一律用 `started_at -> now` 计算

## 三、后台任务中心页面结构

### 1. 一级结构保留三段式

保留：

- 当前任务
- 最近结果
- 历史记录

但重新定义职责：

- 当前任务：解释过程
- 最近结果：支持决策
- 历史记录：仅供回看

### 2. 页头增加同步状态条

页头除“刷新状态”按钮外，应固定显示：

- 自动刷新状态
- 自动刷新频率
- 最近同步时间
- 当前运行中任务数

没有运行中任务时明确表达：

- 当前无自动刷新，仅支持手动刷新

### 3. 当前任务卡的主次顺序

运行中卡片首屏固定表达：

1. 任务标题 + 状态
2. 当前阶段
3. 最近更新
4. 已运行时长
5. 当前可见结果
6. 主动作

### 4. 增加轻量阶段轨迹

运行中卡片顶部增加 4 段轻量轨迹：

- 已提交
- 正在采集
- 正在写入
- 正在收尾

其作用是帮助用户知道“现在在哪”，
不是表达百分比。

### 5. 结果区改为三态

运行中卡片结果区按以下规则切换：

- `collecting` 且无结果指标：
  - 标题：结果将在写入阶段陆续出现
- `persisting / finalizing` 且已有指标：
  - 标题：当前结果
- `success / failed / cancelled`：
  - 标题：本次结果

### 6. 最近结果区强调“下一步”

最近结果区不再承担运行中细节，
每条记录只重点表达：

- 最终状态
- 最终结果
- 失败原因或结论
- 唯一主动作

### 7. 历史记录继续折叠

历史记录保持折叠收纳，不与当前任务抢注意力。

## 四、动作规则

每种状态只给一个主动作：

- 运行中：`终止` 或无动作
- 失败：`按原条件重试`
- 成功：`再次运行`
- 已取消：`继续剩余范围` 或 `按原条件重试`

动作说明可以保留，
但不再让同一卡片出现多个同优先级主按钮。

## 五、异常反馈与文案规则

### 1. 反馈分层

所有反馈统一分为三层：

- 信息态
- 警告态
- 错误态

### 2. 信息态

适用：

- 任务已提交
- 请求已接收，状态稍后刷新
- 终止请求已提交

短句风格：

- 后台已接收请求，状态稍后自动刷新
- 当前处理单元结束后会停止

### 3. 警告态

适用：

- 当前阶段天然没有结果数
- 进度停滞
- freshness 暂时不可用

短句风格：

- 正在采集源站，结果数会在写入阶段出现
- 超过 10 分钟没有新进展，建议先刷新状态

### 4. 错误态

适用：

- 任务提交失败
- 任务执行失败
- 登录失效
- 后台不可用

短句风格：

- 这次处理未完成，可按原条件重试
- 登录状态已失效，请重新登录

### 5. 超时与冲突规则

#### 请求超时

请求超时但后台可能已接单时：

- 归入信息态
- 自动刷新任务中心
- 不显示 error 风格反馈

#### 任务冲突

已有同类任务运行时：

- 归入信息态或警告态
- 反馈为“已有同类任务在运行，已为你刷新当前状态”

### 6. 终止与停滞

#### 终止中

- badge：`正在终止`
- 文案：`当前处理单元结束后会停止`

#### 已终止

- badge：`已终止`
- 文案：`已完成部分结果已保留`

#### 停滞

- 标题：`进度长时间未更新`
- 文案：`先刷新状态；若仍无变化，再考虑重试`

## 六、实施优先级

### Phase 1：最小可信收口

- 收口前台 freshness 文案
- 收口后台提交超时与冲突反馈语义
- 收口任务中心页头同步状态条
- 收口结果区三态文案

### Phase 2：后台过程体验

- 统一阶段字段与状态契约
- 为抓取任务补充 collecting 阶段的可观测指标
- 增加轻量阶段轨迹
- 收口动作优先级与详情层重复信息

### Phase 3：增强项

- 评估是否把后台轮询缩短为更高频率
- 评估是否需要 SSE / WebSocket
- 评估是否引入更明确的发布快照或刷新事件模型

## 七、验收标准

### 后台

- 用户能在首屏知道页面是否自动刷新
- 运行中任务不会再用文案暗示“数字马上会变”
- 当前阶段没有结果数字时，页面会明确说明原因
- 请求超时不会再被感知为“提交失败”
- 冲突、终止中、已终止、停滞、失败的反馈语义彼此清晰区分

### 前台

- 公开前台不再出现伪运行中文案
- freshness 只表达最近一次成功快照
- 不因后台运行中而暴露半成品数据

## 八、边界与风险

### 1. 不应把后台写成说明书

本设计要求：

- 平时少字
- 异常时短句
- 靠结构自解释

而不是通过增加大段说明文字来“教育用户”。

### 2. 抓取任务的真实进展仍受现有链路约束

如果服务层前半段仍然没有可观测指标，
页面只能诚实表达“正在采集”，
不能伪造结果计数。

### 3. 兼容层应在后续版本逐步收口

本期允许保留必要兼容，
但新契约应成为唯一主路径。

## 结论

本期最优方向不是把任务中心做成“更像实时大屏”，
而是把它做成“可信的过程面板”：

- 后台解释过程
- 前台展示成功结果
- 状态契约真实
- 文案短而准
- 异常反馈可执行

只要这五点成立，当前项目的任务体验就会明显提升，
且不会把系统复杂度推到不必要的高度。
