# 后台任务中心 Phase 2 标准版收口设计

## 1. 背景

`2026-04-01` 的任务中心体验设计已经明确了三件事：

- 公开前台继续保持“最近一次成功快照”模型
- 后台任务中心负责解释过程，而不是制造伪实时
- 推送机制、SSE / WebSocket 和更高频同步留到后续增强阶段

当前项目的 Phase 1 已经完成了“最小可信收口”：

- 前台 freshness 文案已经回到成功快照语义
- 后台提交超时、冲突和首次同步状态已经回到中性语义
- 任务中心结果区不再暗示“数字马上会变”

但系统仍有两个明显问题：

1. 后端主契约仍以 `phase + details.metrics` 为核心，缺少稳定的过程字段。
2. 前端任务卡已经更可信，但首屏信息层级仍偏散，解释文案和动作说明偏多。

因此本阶段不再继续做 Phase 1 风格的零散收口，而是进入 Phase 2 的“标准版收口”。

## 2. 设计结论

本阶段采用“标准版收口”，不采用“最小补丁”，也不采用“重型实时化”。

### 2.1 为什么不是最小补丁

如果只补后端 collecting 指标和少量前端文案，确实可以降低实现风险，但用户仍然要靠阅读多段说明文案理解任务流程，任务卡也仍然会出现多个同级动作，体验提升有限。

### 2.2 为什么不是重型实时化

当前系统仍是单机轮询式任务中心，不需要：

- 公开前台运行态
- SSE / WebSocket
- 复杂百分比
- 监控大盘式界面

这些能力会提升实现成本和语义复杂度，却不能显著提高当前招聘信息项目的核心价值。

### 2.3 推荐方案

本阶段的目标收口为一句话：

> 让后台任务中心把过程讲清楚，但不装成实时监控页。

## 3. 目标与非目标

### 3.1 目标

- 建立可稳定消费的后台任务阶段契约
- 让抓取任务在 collecting 阶段具备真实可见的过程指标
- 让任务卡首屏只呈现最关键的过程信息
- 让运行态只保留一个主动作，并让终态后续动作保持少量且语义明确
- 在不引入推送机制的前提下，把后台过程体验做成“可信且易懂”

### 3.2 非目标

- 不为公开前台增加运行态
- 不在前台展示半成品抓取结果
- 不引入 SSE / WebSocket
- 不做复杂图表、实时动画或监控大盘
- 不借本阶段做大规模前端拆分

## 4. 范围边界

### 4.1 纳入本阶段

- 后端任务记录与后台序列化契约
- 抓取任务的 collecting / persisting 可观测指标
- 后台任务卡的阶段轨迹、结果区和动作优先级
- 对应的后端测试、前端 presentation 测试和 UI 契约测试

### 4.2 不纳入本阶段

- 公开前台页面交互增强
- 管理页推送协议
- 通用作业平台化改造
- 任务系统与调度系统的大规模重构

## 4.3 执行约束

- 本阶段凡是涉及 UI 的设计、实现、review 和 polish，均必须使用 `ui-ux-pro-max` skill。
- `ui-ux-pro-max` 用于约束任务卡、阶段轨迹、按钮层级、视觉密度和交互反馈，避免实现阶段只改文案和数据绑定，却忽略信息层级与界面质量。
- 该要求适用于：
  - 后台任务卡首屏结构
  - 阶段轨迹与状态呈现
  - 动作按钮优先级与详情层收口
  - 相关 UI review、验收和回归检查
- 若后续实现拆成多个任务，只要任务涉及 UI 输出，就必须显式声明并执行 `ui-ux-pro-max`。

## 5. 后端设计

### 5.1 任务状态与阶段契约

任务生命周期拆成两层：

- `status`：生命周期状态
- `stage`：运行中阶段

`status` 收口为：

- `queued`
- `running`
- `cancel_requested`
- `success`
- `failed`
- `cancelled`

`stage` 收口为：

- `submitted`
- `collecting`
- `persisting`
- `finalizing`

约束：

- `success / failed / cancelled` 为终态，不再继续暴露运行中 `stage`
- `cancel_requested` 是显式状态，不再只通过 `details.cancel_requested_at` 推断

### 5.2 任务记录主字段

任务记录主路径补齐以下字段：

- `stage`
- `stage_label`
- `stage_started_at`
- `live_metrics`
- `final_metrics`
- `final_summary`

兼容策略：

- `phase`、`details.metrics`、`details.*` 继续保留一个迁移周期
- 后台前端优先消费新字段
- 旧字段只作为兜底，不再作为主契约继续扩展

### 5.3 运行态与完成态分离

运行态只表达“当前已知事实”：

- `status`
- `stage`
- `stage_label`
- `started_at`
- `stage_started_at`
- `heartbeat_at`
- `live_metrics`

完成态只表达“最终结果”：

- `finished_at`
- `duration_ms`
- `final_summary`
- `final_metrics`
- `failure_reason`

额外约束：

- `heartbeat_at` 只能表达“任务仍有心跳”，不能被前端解释为“有新进展”
- 运行中耗时统一由 `started_at -> now` 计算

### 5.4 抓取任务的阶段与指标

抓取任务必须稳定区分：

- `submitted`
- `collecting`
- `persisting`
- `finalizing`

`collecting` 允许暴露：

- `pages_fetched`
- `detail_pages_fetched`
- `raw_items_collected`

`persisting` 允许暴露：

- `posts_seen`
- `posts_created`
- `posts_updated`
- `failures`

规则：

- `collecting` 阶段不伪造结果数
- `persisting` 阶段才展示新增 / 更新结果
- 字段不存在代表当前阶段尚不可得，不等于 `0`

### 5.5 其他任务的最小要求

本阶段不要求所有任务都重做成复杂状态机，但至少满足：

- 每类任务都能映射到统一 `status/stage`
- 每类任务在运行中至少有一组真实 `live_metrics` 或明确的阶段推进
- `base_analysis_backfill` 不能继续只有 heartbeat 没有进展事件

## 6. 前端设计

### 6.1 总体结构

任务中心继续保持三段式：

- 当前任务
- 最近结果
- 历史记录

职责重新强调为：

- 当前任务：解释过程
- 最近结果：支持决策
- 历史记录：仅供回看

页头同步状态条继续保留当前方向，不做重构。

### 6.2 任务卡首屏骨架

任务卡首屏只保留五块：

1. 标题 + 状态 badge
2. 轻量阶段轨迹
3. 最近更新 / 已运行时长
4. 当前结果区
5. 唯一主动作

以下内容不再默认占据首屏主位置：

- 大段 `summaryText`
- 解释性 `modeLabel`
- 动作摘要型说明
- 展开层中的重复解释句

### 6.3 轻量阶段轨迹

运行中卡片增加 4 段轨迹：

- 已提交
- 正在采集
- 正在写入
- 正在收尾

要求：

- 轨迹只表达“当前在哪个阶段”
- 不表达百分比
- 不引入强动效
- 当前阶段高亮，已完成阶段弱完成态，未开始阶段保持静默

### 6.4 结果区规则

结果区标题按阶段切换：

- `collecting` 且无结果指标：`结果将在写入阶段陆续出现`
- `persisting / finalizing` 且已有 `live_metrics`：`当前结果`
- `success / failed / cancelled`：`本次结果`

说明规则：

- 正常状态只保留一句短提示
- 停滞、失败、终止中这类异常态才显示 notice
- 不再用“数量会继续变化”这类泛化承诺覆盖全部运行态

### 6.5 动作优先级

动作优先级按“运行态严格收口，终态保留少量高价值后续动作”处理：

- `running`：`终止`
- `cancel_requested`：无主动作，仅展示终止中状态
- `failed`：`按原条件重试`
- `success`：默认 `再次运行`；若任务支持增量，则补充 `只补剩余`
- `cancelled`：默认 `按原条件重试`；若任务支持增量，则补充 `只补剩余`

约束：

- 运行态保持单一主动作
- `cancel_requested` 不展示主动作
- 终态最多保留 1 到 2 个高价值 follow-up 动作
- 首屏禁止并排展示多个无差别、无层级的 CTA
- 其他低价值变体下沉到详情层，或继续由后端 `actions` contract 控制

## 7. 测试与验收

### 7.1 后端测试

必须补齐：

- `status/stage/stage_started_at/live_metrics/final_metrics` 的持久化与序列化测试
- `cancel_requested` 独立状态测试
- 抓取任务 `submitted -> collecting -> persisting -> finalizing -> success/failed/cancelled` 阶段序列测试
- collecting 只暴露采集指标、persisting 才暴露结果指标的测试

### 7.2 前端测试

必须补齐：

- 阶段轨迹渲染测试
- 结果区按 `stage/live_metrics` 切换测试
- 动作优先级与终态后续动作测试
- 新旧契约并存期间的优先级测试

### 7.3 最终验收口径

本阶段完成时，必须满足：

- 用户能一眼判断任务现在处于哪个阶段
- collecting 阶段不会被误判为“卡住”或“结果应该已经出来”
- 任务卡不会再堆叠过多解释性文案
- 运行态只保留一个主动作，终态只保留少量高价值后续动作
- 前台继续只展示最近一次成功快照

## 8. 实施顺序

1. 先改后端契约与取消状态
2. 再补抓取 collecting 指标与阶段序列
3. 然后在 `ui-ux-pro-max` 约束下改前端任务卡与动作优先级
4. 最后补测试并更新交付口径文档

## 9. 风险与取舍

### 9.1 心跳不等于进展

如果前端继续把 `heartbeat_at` 当成“有新结果”，仍会制造伪实时感。本阶段必须把这层语义切开。

### 9.2 契约迁移期间容易混读旧字段

如果前端继续无条件优先读 `phase/details.*`，即使后端新增了新字段，产品语义也不会真正切换。本阶段必须把“新字段优先、旧字段兜底”写进实现和测试。

### 9.3 不能先做 UI，再补 collecting 指标

如果前端先上阶段轨迹，但后端 collecting 仍没有真实可观测指标，最终只会把“看起来更像实时”做得更重，不会更真。

## 10. 结论

Phase 2 不应该继续做零散修补，也不应该跳到推送式实时架构。

最合理的路径是：

- 用最小必要的后端契约收口，建立真实阶段模型
- 用轻量但稳定的前端结构，把任务过程讲清楚
- 继续保持公开前台的成功快照模型

这套方案的完成标准不是“更像实时”，而是“更真实、更易懂、更像正常项目”。
