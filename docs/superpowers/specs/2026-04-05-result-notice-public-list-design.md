# 结果公示从默认前台列表收口设计

## 1. 背景

当前公开前台列表页标题和用户心智更接近“辅导员招聘公告阅读页”，但系统默认查询口径实际是：

- 只要是辅导员相关内容就会进入列表
- 未显式选择 `event_type` 时，不会自动限制为 `招聘公告`
- 因此 `结果公示` 会和 `招聘公告` 一起出现在默认第一页

这带来两个体验问题：

1. 默认页标题写的是“辅导员招聘公告”，但首页会混入大量拟聘用名单公示。
2. 结果公示经常显示 `岗位待整理`，语义像“系统还没整理完岗位”，但这类内容本质上是结果披露，不应该暗示还要继续抽取岗位。

仓库当前已明确承认一个事实：

- 结果公示类帖子的岗位口径仍在持续收口

因此本次不是去“提升结果公示的岗位抽取能力”，而是先把公开前台的默认语义收正。

## 2. 设计结论

本次采用“默认页只看招聘公告，结果公示保留为可筛选内容”的收口方案。

一句话结论：

> 默认公开前台不再展示结果公示；结果公示仍保留在公告类型筛选里可单独查看；结果公示不再显示 `岗位待整理`。

## 3. 目标与非目标

### 3.1 目标

- 让默认前台列表与“招聘公告”心智一致
- 保留结果公示的可访问性，但改为显式筛选进入
- 移除结果公示上的错误状态暗示，避免“岗位待整理”误导用户
- 保持列表接口、统计接口、事件类型下拉选项的口径一致

### 3.2 非目标

- 不删除或下线结果公示数据
- 不修改后台抓取、AI 分类、结果公示归类逻辑
- 不在本次引入新的公告类型分组体系
- 不借机重做列表页视觉结构

## 4. 用户心智与交互原则

公开前台应保持目录型产品的最短路径：

- 默认打开就看到“还值得继续点击的招聘公告”
- 已结束、已公示的结果类信息不抢占首页注意力
- 用户若确实要查公示，可以通过现有“公告类型”筛选显式进入

本次沿用现有筛选结构，不新增额外入口，不增加新的切换负担。

## 5. 方案对比

### 5.1 方案 A：默认页直接排除结果公示，并保留筛选入口

优点：

- 最符合当前页面标题和用户预期
- 首页结果质量提升最直接
- 不需要新增新的交互组件

代价：

- 默认统计数字会下降
- 列表、统计和下拉选项都要跟着统一口径

### 5.2 方案 B：默认页继续展示结果公示，但改文案

优点：

- 改动最小

缺点：

- 页面标题、列表价值和用户意图仍不一致
- 首页仍会被大量公示挤占

### 5.3 方案 C：默认页保留结果公示，但降低权重或弱化展示

优点：

- 不丢默认曝光

缺点：

- 规则更复杂
- 用户仍然会问“为什么首页有公示”
- 不能真正解决默认口径歧义

推荐采用方案 A。

## 6. 详细设计

### 6.1 默认查询口径

当前前台默认范围仍保持：

- 招聘范围：`所有辅导员相关`

但新增一条隐式默认规则：

- 当用户没有显式选择“公告类型”时，前台默认按 `event_type=招聘公告` 请求列表和统计

一旦用户显式选择公告类型：

- 选择 `结果公示`：显示结果公示
- 选择 `招聘公告`：显示招聘公告
- 选择其他已有类型：按用户选择展示

这意味着：

- 默认首页只看招聘公告
- 公告类型筛选仍然完整保留
- “招聘范围”和“公告类型”继续是两个独立维度

### 6.2 结果公示的完整度语义

当前 `岗位待整理` 属于通用“辅导员相关但没有可展示岗位”的状态。

本次对 `结果公示` 单独收口为：

- 结果公示不再进入 `jobs pending` 语义
- 即使没有岗位结构化结果，也不显示 `岗位待整理`
- 结果公示仍可显示：
  - `AI 整理` 或 `规则整理`
  - `已收录正文`
  - `结果公示`

原因：

- 结果公示的核心价值是名单与结果披露，不是岗位检索
- 这类帖子常见附件和正文内容以人名、名单、聘用结果为主
- 对用户来说，“没有岗位结构化结果”不代表“系统还有待完成的工作”

### 6.3 列表、统计与筛选项一致性

以下三处必须统一默认口径：

- 列表数据请求
- 顶部统计卡数字
- 公告类型下拉选项的数据来源

约束如下：

- 当用户未显式选择 `event_type` 时，这三处都按 `招聘公告` 统计
- 公告类型下拉中仍应包含 `结果公示`，以便用户切换
- 如果当前选择的是 `结果公示`，列表和统计都切到 `结果公示` 口径

换句话说：

- 默认口径要收紧
- 可选口径不能丢失

### 6.4 文案语义

本次不要求重做标题，但要求状态文案不再误导：

- `岗位待整理` 只用于真正仍应继续做岗位抽取的招聘公告
- 不得再用于 `结果公示`

如后续仍需给结果公示补一类状态，只能使用归档式、中性语义，例如：

- `结果公示`
- `名单公示`
- `结果归档`

本次优先方案是不新增状态，而是直接移除 `岗位待整理`。

## 7. 后端与前端影响面

### 7.1 前端

- 默认构造列表请求参数时，需要把“未选择公告类型”解释为默认 `招聘公告`
- 默认构造统计请求参数、事件类型选项请求参数时，也要保持同口径
- 列表卡片与详情页的完整度展示，需要在 `结果公示` 下屏蔽 `岗位待整理`

### 7.2 后端

- 不要求改变结果公示的归类、摘要和 `should_track` 逻辑
- 如当前 `record_completeness.jobs` 的生成逻辑无法区分结果公示，需要补一个稳定分支，让前端拿到“不适用”或至少不再拿到 `pending`

后端是否新增新的 `jobs` 状态值，可由实现阶段决定：

- 方案一：后端新增 `not_applicable`
- 方案二：后端对结果公示直接返回 `missing` 或其他不会触发前端警告态的值

实现阶段应优先选改动最小、表达最清晰的一种。

## 8. 测试要求

至少补齐以下验证：

- 默认进入列表页时，请求参数隐式带上 `招聘公告` 口径
- 默认统计卡与列表总数保持一致
- 公告类型下拉仍能提供 `结果公示` 选项
- 手动切换到 `结果公示` 后，列表可正常展示结果公示数据
- 结果公示卡片不再出现 `岗位待整理`
- 普通招聘公告在确实缺岗位结构化结果时，仍可显示 `岗位待整理`

## 9. 风险与权衡

### 9.1 风险

- 默认统计数字会明显变化，可能与当前线上截图不一致
- 如果事件类型下拉完全依赖“当前默认口径的统计分布”，可能导致 `结果公示` 选项消失
- 如果只改前端不改完整度状态，详情页和列表页可能出现口径不一致

### 9.2 应对

- 实现时优先检查“事件类型下拉是否仍可切到结果公示”
- 列表页和详情页共用同一套结果公示完整度口径
- 保持改动聚焦，不扩展到后台任务、抓取和 AI 提示词

## 10. 验收标准

满足以下条件即可视为本次收口完成：

1. 默认首页不再出现结果公示
2. 用户通过“公告类型 -> 结果公示”仍可查看结果公示
3. 结果公示卡片和详情页都不再显示 `岗位待整理`
4. 招聘公告原有的 `岗位待整理` 提示不被误伤
5. 列表、统计和筛选项默认口径一致
