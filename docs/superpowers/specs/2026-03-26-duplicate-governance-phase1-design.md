# 重复治理一期设计

## 背景

当前项目对帖子去重基本只依赖 `canonical_url`。这能挡住“同一个链接重复抓取”，但挡不住“同一篇内容换了另一个链接”这种更常见的线上重复。

这会直接带来几个问题：

- 前台列表会出现看起来一模一样的公告，影响使用体验。
- 统计摘要会被重复帖子放大，结果不准。
- 后续的 AI 分析、统计字段抽取、岗位抽取会重复消耗资源。
- 管理页无法判断哪些是历史脏数据，哪些是真的新公告。

这期目标不是“删库式清洗”，而是做一套保守、可回滚、先能用的软去重方案。

## 目标

- 识别高置信重复帖子，并把它们归并到同一个重复组。
- 每个重复组只保留 1 条主记录给前台和统计使用。
- 默认隐藏重复记录，但不物理删除数据。
- 给管理页补一个重复治理摘要，方便检查历史脏数据。
- 把去重触发接到抓取和启动补齐流程里，避免新重复继续堆积。

## 非目标

- 不做物理删除。
- 不做跨来源站点的激进合并。
- 不做岗位跨帖子去重。
- 不做人工审核流和手动合并/拆分后台。
- 不做复杂相似度模型或向量检索。

## 总体方案

采用“保守软去重”。

核心思路：

1. 给帖子补去重元数据，不删原始记录。
2. 抓取完成后，对本次涉及的帖子跑一次增量去重。
3. 启动时补跑一次历史去重，先把旧数据收住。
4. 前台列表、详情页、统计摘要默认只认主记录。
5. 管理页新增重复治理摘要，能看到重复组规模和命中原因。

这样能优先解决“用户看到重复公告”这个痛点，同时把误伤风险压到最低。

## 数据模型

在 `posts` 表上新增以下字段：

- `duplicate_status`
  - 值域：`primary` / `duplicate` / `none`
  - 默认 `none`
  - 含义：
    - `primary`：重复组主记录
    - `duplicate`：重复组中的从记录
    - `none`：未进入重复组

- `duplicate_group_key`
  - 字符串
  - 同一个重复组内相同
  - 用来聚合和展示

- `primary_post_id`
  - 可空外键，指向主记录
  - 主记录自己为空
  - 从记录指向主记录

- `duplicate_reason`
  - 字符串
  - 记录命中的高置信规则
  - 例如：
    - `canonical_url`
    - `original_url`
    - `source_date_title`
    - `source_date_title_content_fingerprint`

- `duplicate_checked_at`
  - 最近一次完成去重判定的时间

说明：

- 这期不单独建新表，先把字段放在 `posts` 上，落地更快。
- 后面如果需要做人工审核或历史比对，再拆 `post_duplicates` 之类的表。

## 判重规则

只认高置信重复，命中以下任一规则才归组。

### 规则 1：`canonical_url` 相同

- 这是现有系统已经部分依赖的规则。
- 命中后直接归同组。
- `duplicate_reason = canonical_url`

### 规则 2：`original_url` 相同

- 用来补抓取链路中链接规范化不稳定的情况。
- 同一来源下如果 `original_url` 一致，也直接归组。
- `duplicate_reason = original_url`

### 规则 3：同来源 + 同发布日期 + 规范化标题一致

命中条件：

- `source_id` 相同
- `publish_date` 在同一天
- 标题规范化后完全一致

标题规范化只做保守清洗：

- 去首尾空白和多余空格
- 统一全角/半角空格
- 统一常见中文/英文标点差异
- 不做激进同义词替换
- 不改语义词序

`duplicate_reason = source_date_title`

### 规则 4：同来源 + 标题一致 + 正文指纹一致

只有在前 3 条没命中，但仍高度可疑时才使用。

命中条件：

- `source_id` 相同
- `publish_date` 同一天，或相差不超过 1 天
- 标题规范化后完全一致
- 正文清洗后生成内容指纹，指纹一致

正文清洗复用现有正文归一化思路：

- 去明显站点噪音
- 统一空白和换行
- 去重复拼接尾段
- 去附件尾部噪声

`duplicate_reason = source_date_title_content_fingerprint`

## 明确不判重复的情况

- 只因为标题相似，但正文不同。
- 只因为学校名相同。
- 只因为岗位表相似。
- 不同来源站点之间的内容相似。
- 招聘公告和后续结果公示、考试通知、资格复审通知之间的关联。

这部分故意保守，避免误把流程性后续公告压没。

## 主记录选择规则

重复组里需要选 1 条主记录。

优先级从高到低：

1. 有正文
2. 有附件
3. 结构化字段更多
4. 岗位结果更多
5. 分析/统计结果更完整
6. `created_at` 更早
7. `id` 更小

这样能尽量保留“信息最全、最稳定”的那条给前台显示。

## 前台行为

### 列表页

- 默认只展示：
  - `duplicate_status = primary`
  - 或 `duplicate_status = none`
- `duplicate_status = duplicate` 的记录默认隐藏

### 详情页

- 如果访问的是主记录：
  - 正常展示
  - 页面可提示“该公告已归并重复来源”

- 如果访问的是从记录：
  - 后端直接返回主记录
  - 或返回一个明确的主记录跳转信息

这期固定采用“后端直接返回主记录内容”，前端不用再多一套跳转状态机。

### 统计摘要

- 前台 `/posts/stats/summary` 只统计主记录和未归组记录
- 否则总帖子数、近几天新增、事件分布都会被重复帖子抬高

## 管理页行为

管理页新增一个“重复治理”摘要区块，先做只读，不做人工操作台。

展示内容：

- 重复组数
- 被折叠帖子数
- 当前主记录数
- 未归组记录数
- 最近一次去重时间

再给一个最近重复组列表，字段包括：

- 主记录标题
- 组内帖子数
- 重复原因
- 来源
- 发布日期

这期不做：

- 手动指定主记录
- 手动拆组
- 一键删除从记录

## 触发时机

### 抓取完成后增量触发

- `scrape_and_save()` 在保存/更新完帖子后，收集本轮涉及的帖子 ID
- 对这些帖子跑一次增量去重

### 启动时历史补齐

- 启动初始化时补跑一次历史去重
- 只扫描现有帖子，不依赖重新抓取

### 管理页手动触发

- 这期不做“重跑重复治理”按钮
- 这期只做重复摘要和最近重复组列表

## 服务边界

新增一个独立服务模块，避免把重复治理继续塞进已有大文件。

模块职责：

- `duplicate_service.py`
  - 标题规范化
  - 正文指纹生成
  - 重复候选查找
  - 主记录选择
  - 写回帖子去重字段
  - 重复摘要统计

现有模块只做接线：

- `scraper_service.py`
  - 抓取后调用增量去重

- `bootstrap.py`
  - 启动时调用历史去重补齐

- `posts.py`
  - 列表、详情、统计摘要默认隐藏从记录

- `admin.py`
  - 暴露重复摘要接口

## API 设计

新增管理端摘要接口：

- `GET /api/admin/duplicate-summary`

返回结构：

- `overview`
  - `duplicate_groups`
  - `duplicate_posts`
  - `primary_posts`
  - `unchecked_posts`

- `reason_distribution`
  - 按 `duplicate_reason` 分布

- `latest_checked_at`

- `latest_groups`
  - 最近若干个重复组摘要

这期不新增前台去重接口，前台直接吃现有列表/详情接口的新口径。

## 实施顺序

一期实现顺序如下：

1. 扩 `Post` 模型和数据库补齐逻辑
2. 新增 `duplicate_service`
3. 给抓取链路接增量去重
4. 给启动链路接历史补齐
5. 调整前台列表、详情、统计摘要查询口径
6. 新增管理页重复摘要接口
7. 最后再补管理页展示

这样每一步都能单独验证，不会一上来改炸全局。

## 测试策略

至少覆盖这些场景：

### 后端服务测试

- 相同 `canonical_url` 会归为一组
- 相同 `original_url` 会归为一组
- 同来源同日期同标题会归为一组
- 标题一致但正文不同，不归组
- 不同来源相似标题，不归组
- 主记录选择会优先选信息更完整的帖子

### 接口测试

- 列表页默认隐藏从记录
- 详情页访问从记录时返回主记录
- 统计摘要只统计主记录
- 管理页重复摘要返回正确计数

### 回归测试

- 抓取新帖子后，非重复帖子不受影响
- AI 分析、统计字段、岗位抽取在主记录上还能正常工作
- 历史数据补齐不会把明显不同的帖子误合并

## 风险与控制

### 风险 1：误伤流程性公告

控制方式：

- 严格限制在同来源内判重
- 标题必须规范化后一致
- 正文指纹规则只作为最后一层高门槛兜底

### 风险 2：主记录选错

控制方式：

- 用“信息完整度优先”的固定排序
- 不做随机或模糊选择

### 风险 3：历史脏数据一次处理太多

控制方式：

- 先做保守规则
- 管理页先给摘要，方便抽查结果
- 不物理删除，出问题还能重新选主记录

## 结论

这期最合适的是“保守软去重”。

它能解决当前最影响体验的重复帖子问题，又不会把系统带进高风险的大清洗。先把重复组、主记录、前台隐藏和管理页摘要做起来，后面如果还要继续收严，再做二期人工干预和更复杂的相似度判断。
