# 整站 UI 与 Design System 升级设计

## 背景

当前项目的前台列表、详情页和后台管理台已经具备可演示能力，但整体界面仍处在“局部修过、全局未收口”的状态。

本次工作不再继续做零散修补，而是按“设计系统 -> 后台任务中心 -> 详情页 -> 其余页面”的顺序，重新建立整站 UI 规则，并在现有 Vue + Tailwind + Reka UI 基础上逐步落地。

如旧设计稿、旧 plan 与当前代码不一致，以当前代码实际行为、README 和 STATUS 的现口径为准。

## 审计结论

### 1. 样式系统已经失控，但还没到必须推倒重来

- `frontend/src` 中存在大量直接写在页面里的视觉类名，颜色、圆角、阴影、面板背景和间距重复散落。
- `rounded-[28px]`、`rounded-2xl`、`bg-white/90`、`bg-slate-50`、`border-slate-200` 一类表达在多个页面和组件层重复，说明当前 token 没有真正成为页面实现依据。
- 共享组件已经存在，但更多是在复用“白底圆角卡片外壳”，不是在复用稳定的视觉语义和信息结构。

### 2. 前台和后台用了同一套“软白卡片模板”，上下文区分不够

- 列表页、详情页、管理页都使用了类似的浅蓝背景、白色大卡片、重圆角和轻阴影。
- 前台缺少更明确的阅读层级；后台缺少更强的操作区与结果区区分，视觉上都偏“温和”，不够“可信、克制、可扫描”。

### 3. 招聘详情页首屏仍然在传达“数据块”，而不是“判断信息”

- 详情页虽然已经把内部来源说明折叠到了下方，但首屏仍按 `hero -> facts card -> jobs card -> attachments` 线性堆叠，结构还是偏“模块陈列”。
- 多岗位公告的首屏事实区当前仍然优先取第一条岗位数据，容易把“第一条岗位”误当成“整篇公告的主结论”。
- 大段正文仍然在视觉上压过前面摘要层，导致“核心判断信息”不够稳定。

### 4. 后台任务中心的语义已经比上一版真实，但表现层仍偏重、偏散

- 任务状态、`stage_only / determinate`、动作语义、`rerun_of_task_id` 已经具备较好的基础。
- 当前问题不再主要是术语错误，而是展示结构过多：总览卡、统计卡、任务卡、说明卡、操作说明卡、详情卡层层嵌套，阅读阻力仍然偏高。
- “当前阶段 / 结果统计 / 可执行动作 / 失败原因”虽然都有了，但没有形成稳定的视觉主次。

### 5. 表单和反馈语义已经开始统一，但交互层还没形成体系

- Tabs 已经接入 Reka UI，但折叠、选择、提示、对话、进度等 primitive 还没有形成统一使用边界。
- 很多表单控件仍是页面内直接写样式，焦点态、尺寸、密度和帮助信息格式不统一。

### 6. 文案治理已有成果，但仍需要把“少写字、靠结构表达”继续推进

- 明显的开发者口吻、内部说明型文案大部分已经移除。
- 但不少页面仍依赖说明句来解释界面结构，说明结构本身还不够自解释。

## 设计目标

- 建立一套真正可复用的 design token、surface 和信息组件层。
- 让前台更像“高效阅读与判断页面”，让后台更像“可信的处理工作台”。
- 强化任务状态、结果统计、失败原因和动作边界的视觉语义。
- 继续保持用户视角文案，避免内部术语重新回流到最终界面。
- 按阶段迁移，不要求第一步全站重写。

## 设计原则

### 1. 可信优先于花哨

- 不用伪进度、不用装饰性统计、不用过度动画。
- 信息密度可以高，但必须可扫描、可核对、可理解。

### 2. 用结构代替解释

- 优先用区块顺序、标题、标签、状态色、分组、表格和 definition list 建立阅读路径。
- 仅在结构无法自解释时补一句短说明。

### 3. 前台偏阅读，后台偏操作

- 前台强调“是否值得继续关注”。
- 后台强调“现在在做什么、做到哪一步、下一步能做什么”。

### 4. 共享的是规则，不是统一模板

- 共享 token、状态语义、表单尺寸、按钮层级、surface 语言。
- 不强行让前台和后台长得一模一样。

### 5. Reka UI 只承担 primitive，不接管业务页面表达

- 使用 Reka UI 解决可访问、状态管理和复合交互。
- 页面布局、业务卡片、阅读层级、任务语义仍由项目组件层负责。

## 视觉方向

整体方向定为：`编辑式 + 工具化 + 克制的公共服务感`

- 气质：可信、平静、专业、节制
- 关键词：editorial rhythm、structured utility、clear state hierarchy
- 避免：发光渐变、过度玻璃态、全站超大圆角、过量 KPI 卡片、为了“高级感”牺牲信息清晰度

## 信息架构调整

## 前台

### 列表页

- 保留搜索和筛选，但压缩首屏垂直高度。
- 统计区从“大卡片阵列”调整为更紧凑的 summary strip / concise stats。
- 列表卡片减少无效胶囊和浅色块重复，强化标题、岗位概览和关键信息的主次。

### 详情页

首屏改成：

1. 标题与来源
2. 核心判断摘要
3. 关键事实
4. 岗位明细

具体规则：

- 核心判断摘要不再简单复用第一条岗位字段。
- 多岗位公告优先表达“岗位数、人数范围、学历/专业差异、报名时间”等整篇维度事实。
- “更新说明 / 信息说明 / 原文来源逻辑”全部降到次级层。
- 附件保留，但不抢首屏主次。

## 后台

保留当前一级结构：

- `总览`
- `处理任务`
- `任务中心`
- `系统设置`

但重做页面骨架：

- 总览：压缩为系统状态 + 最近结果 + 待处理重点，减少二级解释块。
- 处理任务：减少嵌套卡，强化每个任务组的目标、当前结果、主操作。
- 任务中心：重做为“当前任务 / 最近结果 / 历史记录”三段式版面，重点突出阶段、结果、操作和失败原因。
- 系统设置：回归配置页，不再承载多余状态叙事。

## Reka UI 使用边界

### 适合引入 Reka primitive 的场景

- `Tabs`
  - 后台一级导航、处理任务二级切换
- `Accordion / Collapsible`
  - 信息说明、任务详情、筛选高级项
- `Select`
  - 数据源、范围选择、批量设置
- `Progress`
  - 仅用于 `determinate` 且总量可信的任务
- `Tooltip`
  - 按钮禁用原因、状态解释
- `Dialog`
  - 高风险操作确认，不用于普通说明
- `Toast`
  - 提交成功 / 失败反馈

### 应保留并重构的现有组件

- `AppPageHeader`
- `AppSectionHeader`
- `AppStatusBadge`
- `AppNotice`
- `AppStatCard`
- `AdminTaskActionCard`
- `AdminTaskRunCard`
- `PostHeroSection`
- `PostFactsSection`
- `PostJobsSection`

原因：

- 这些组件已经承载当前业务语义，直接删掉收益不高。
- 但必须收口样式、密度、标题层级和信息表达方式。

## 页面迁移顺序

### 阶段 1：设计系统基础层

- 重建全局 token
- 重构共享 UI primitives
- 收口公共 surface、按钮、状态 badge、notice、tabs、disclosure
- 统一前后台 page shell / section shell

### 阶段 2：后台任务中心

- 任务中心重排为“当前任务 / 最近结果 / 历史记录”三段式。
- 任务卡片主次稳定为“阶段 -> 结果 -> 动作 -> 失败原因 / 详情”。
- 保留现有重试 / 重跑 / 补剩余 / 重新抓取语义边界，不再回退到模糊动作文案。

### 阶段 3：招聘详情页

- 首屏改为围绕核心判断信息组织，不再由第一条岗位字段主导整篇公告结论。
- 关键事实和岗位明细分层呈现，正文与附件降到次级层级。
- 增加快捷跳转与长内容阅读辅助，减少大段内容对首屏判断的干扰。

### 阶段 4：列表页与后台剩余页面

- 列表页首屏、筛选区、结果卡与空状态已经按统一 token / primitive 收口。
- 后台总览、处理任务、系统设置与 404 页已经统一到同一套页面骨架和反馈语义。

### 阶段 5：响应式与可访问性回归

- `768 / 1024` 中断点下，后台处理页改为更早进入双列与紧凑头部布局，避免页面长度被无意义单列放大。
- 系统设置页在中断点下为长数据源字段和调度表单重新分配宽度，避免配置确认时被截断。
- 任务中心在中断点下改用更轻的空态提示，并让结果卡更早进入双栏信息布局，减少纵向堆叠。
- 首页筛选区和详情页首屏在 `1024px` 下提前进入横向布局，减少首屏无效高度。
- `AppDisclosure`、`AppTabNav`、`AppFactList`、详情页锚点导航等共享组件补齐了可访问性与长内容处理。
- 回归重点不再是新增视觉风格，而是守住信息层级、密度、状态语义和键盘可达性。

## 实际落地结果（2026-03-31）

### 已落地的 shared design system

- 全局 design tokens 已经覆盖颜色、状态色、间距、圆角、阴影、字号、层级和基础 surface 语义。
- `AppSurface / AppPageHeader / AppSectionHeader / AppStatusBadge / AppNotice / AppActionButton / AppTabNav / AppDisclosure / AppMetricPill / AppEmptyState / AppFactList / AppSectionNav` 已成为当前站点的共享 UI 基础层。

### 已落地的前台结果

- 列表页首屏压缩为更易扫描的摘要、筛选与结果节奏。
- 招聘详情页首屏已经聚焦“值不值得继续看”的判断信息，而不是堆叠结构化数据块。
- 附件、原文正文、来源说明与补充信息已稳定落到次级层级。

### 已落地的后台结果

- 总览页聚焦系统状态、整理能力与待处理重点。
- 处理任务页围绕“任务组 -> 当前结果 -> 主动作 -> 范围设置”组织，并按基础处理 / 智能整理分层。
- 任务中心已经把进行中任务、最近结果和历史记录拆开，避免不同状态混堆，并在 `768 / 1024` 下压缩空态和结果卡高度。
- 系统设置页回归配置管理本身，不再承载冗余状态叙事，并在 `768 / 1024` 下保留足够的字段宽度与表单可读性。

### 仍需持续关注的回归点

- 新增任务类型时，必须继续遵守当前动作语义边界，避免重新混淆“重试 / 重跑 / 补剩余 / 重新抓取”。
- 新增详情字段时，必须优先判断其属于“首屏核心判断”还是“次级补充信息”，不要重新把大块结构化字段推回首屏。
- 新增组件优先复用 shared primitive；只有业务结构真正不同，才新增模块级组件。

## 风险点

- 共享 primitive 改动会影响前后台多个页面，需要逐步替换并持续构建验证。
- 详情页事实层改造涉及字段聚合逻辑，需要避免把“岗位级字段”误写成“公告级结论”。
- 后台任务中心若只改样式不改信息主次，会继续保留阅读成本。
- Reka primitive 的引入要控制边界，避免页面层被过度组件库化。

## 验证方式

- 前端 `npm test`
- 前端 `npm run build`
- Playwright 目检：
  - `/`
  - `/post/:id`
  - `/admin`
- 真实接口核对：
  - `/api/posts`
  - `/api/posts/:id`
  - `/api/admin/task-runs`
  - `/api/admin/task-runs/summary`

## 结论

当前项目的问题不再是“有没有重构过”，而是“已经有一批半成品规则，但还没有成为整站统一规则”。

本次应以 `设计系统基础层 -> 后台任务中心 -> 详情页 -> 其余页面` 的顺序推进，先把规则立住，再逐页替换旧外壳和旧层级。
